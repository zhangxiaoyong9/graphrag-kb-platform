# Logging and observability

KB Platform uses Python's standard `logging` package across the server, worker,
and MCP processes. Logs go to stderr and to rotating gzip files under `logs/`
by default.

## Correlation model

The formatter adds fields when they are available:

```text
service → request → query → kb → job → step → unit → llm_call → operation
```

- `service`: `server`, `worker`, or `mcp`.
- `request`: accepted from a valid `X-Request-ID` header or generated by the API.
- `query`: one streaming query or conversation turn.
- `job` / `step` / `unit`: indexing control-plane identities.
- `llm_call`: one completion stream, non-stream completion, or embedding collection.
- `operation`: `completion`, `completion_stream`, or `embedding`.

The API returns `X-Request-ID`. Supply the same header from a reverse proxy or
client to correlate upstream and platform logs.

## LLM events

Completion calls emit safe metadata only; prompts and answers are not logged.

| Event | Important fields |
|---|---|
| `llm.start` | provider, model, safe endpoint, attempt, message/input size and hash |
| `llm.success` | latency, stream flag, token counts, output length |
| `llm.error` | status, retriable flag, upstream request ID, redacted response excerpt |
| `llm.transport_error` | exception type, latency, redacted message |
| `llm.stream_error` | parser/provider stream failure |
| `llm.invalid_response` | content type and redacted response excerpt |
| `llm.exhausted` | profile count, total latency, last redacted error |
| `llm.structured_parse_error` | response length and validation error type |
| `llm.probe_result` | health probe status and latency (`DEBUG`) |

## Embedding events

Embedding is asynchronous end to end. It retries 429, 5xx, timeout, and
transport failures. `KB_EMBED_MAX_ATTEMPTS` controls the per-batch attempt cap
(default `3`).

| Event | Important fields |
|---|---|
| `embedding.start` | provider, model, item count, batches, input size/hash |
| `embedding.batch_start` | batch and attempt progress, safe endpoint |
| `embedding.batch_success` | items, dimensions, tokens, latency |
| `embedding.http_error` | status, retry decision, redacted response excerpt |
| `embedding.transport_error` | exception type, attempt, latency |
| `embedding.invalid_response` | content type and redacted response excerpt |
| `embedding.vector_count_mismatch` | expected and actual vectors |
| `embedding.done` | vectors, dimensions, tokens, total latency |
| `embedding.collection_*` | parquet collection-to-vector-store lifecycle |

Input hashes are diagnostic fingerprints, not cache keys. They allow operators
to tell whether two failures used the same input without recording private text.

## Module audit

| Area | Logged boundaries | Intentionally omitted |
|---|---|---|
| API middleware/routes | request lifecycle, status-sensitive levels, mutations, export | bodies, credentials, cookies |
| Query/conversation | query lifecycle, first token, engine selection, rewrite fallback | query text and answer text |
| Worker/engine | job, step, unit, recovery, chunking, embedding collections | heartbeat ticks and every dataframe row |
| LLM | attempts, success, tokens, failover, breaker/probe transitions, parse errors | prompts, raw successful responses, keys |
| Embedding | collection/batch progress, retries, validation, tokens/dimensions | text and vectors |
| Neo4j | driver lifecycle, query/execute timing, row counts, truncation | passwords, full Cypher parameters, vector values |
| Vector store | index upserts and item counts | vector payloads |
| Realtime | subscription lifecycle and isolated poll failures | every poll and websocket frame |
| MCP | HTTP/query lifecycle, duration and failures | document/query contents |
| Document input | parser success and fallback reason | extracted content |
| DB/crypto/models | failures surface at service/route boundaries | SQL rows, encrypted/plaintext secrets |
| Install CLI | user-facing command results | separate persistent log pipeline |

Low-level value objects, enums, Pydantic models, pure formatting functions, and
package `__init__` modules deliberately do not log. Logging those modules would
add noise without creating an actionable lifecycle boundary.

## Configuration

| Variable | Default | Purpose |
|---|---|---|
| `KB_LOG_LEVEL` | `INFO` | root level |
| `KB_LOG_LEVELS` | empty | per-logger overrides |
| `KB_LOG_DIR` | `logs` | file output directory |
| `KB_LOG_CONSOLE` | `true` | stderr output |
| `KB_LOG_ROTATE_WHEN` | process-specific | rotation unit |
| `KB_LOG_ROTATE_INTERVAL` | process-specific | rotation interval |
| `KB_LOG_ROTATE_BACKUP_COUNT` | process-specific | retained archives |
| `KB_EMBED_MAX_ATTEMPTS` | `3` | attempts per embedding batch |

Examples:

```bash
KB_LOG_LEVEL=DEBUG \
KB_LOG_LEVELS=kb_platform.llm=DEBUG,kb_platform.engine.unit_worker=INFO \
uv run python -m kb_platform.worker kb.db
```

```bash
rg 'llm_call=emb-|embedding\.' logs/worker.log
rg 'request=<id>|query=<id>' logs/server.log
```

## Redaction

The final formatter removes common `Authorization: Bearer`, `api_key`,
`x-api-key`, and `sk-...` credential forms. Error response excerpts are
single-line, length-bounded, and redacted before logging. Redaction is a final
safety net: call sites must still avoid logging complete request/response bodies.
